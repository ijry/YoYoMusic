# 悠悠乐听自动连播收口设计

**日期:** 2026-07-05  
**项目:** 悠悠乐听音乐播放器  
**主题:** 自动连播与播放结束状态同步

## 1. 背景

当前项目已经具备本地文件/文件夹导入、`rodio` 音频播放、主窗口状态接入、迷你播放器窗口和桌面歌词窗口基础路由，但自动连播仍未真正收口。

现阶段的主要问题是：

- 自然播放结束后没有统一的 Rust 侧推进逻辑。
- `PlaybackState`、`Playlist.currentIndex` 和多窗口显示之间仍以前端轮询为主，缺少结束事件驱动。
- 顺序播放、列表循环、单曲循环、随机播放的边界规则没有完全固化到同一条状态更新链路中。

本设计只解决自动连播与其直接依赖的状态同步问题，不扩展到托盘快捷键、真实歌词滚动或新的持久化结构。

## 2. 目标

本轮设计目标：

- 在 Rust Core 中完成“自然播放结束 -> 按播放模式推进或停止”的统一处理。
- 保证 `PlaybackState` 与 `Playlist.currentIndex` 始终一起变化，不出现状态分裂。
- 在自然切歌后向所有窗口广播统一事件，使主窗口和迷你播放器立即更新。
- 保留当前前端轮询作为兜底机制，但不再依赖前端判断是否该自动切歌。

## 3. 非目标

本轮明确不做：

- 托盘菜单和全局快捷键的真实播放联动收口。
- 桌面歌词文本随歌词时间轴滚动。
- 将现有前端轮询完全移除。
- 复杂随机历史算法或“最近播放队列”。
- 设置、会话、播放列表结构的持久化重构。

## 4. 方案选择

最终选择：**Rust 事件驱动自动连播**。

不采用前端驱动方案的原因：

- 前端不应成为播放结束判断源，这会削弱 Rust Core 作为唯一状态真相源的架构。
- 当主窗口未打开、失焦、卡顿或未来存在更多窗口时，前端驱动天然不可靠。

不采用纯 Rust 轮询但不广播事件的原因：

- 只在 Rust 内推进曲目会让主窗口、迷你播放器、桌面歌词继续依赖高频轮询，联动滞后。

采用 Rust 事件驱动后，播放结束和用户手动切歌都可以复用同一条状态更新链路，后续托盘和快捷键也能继续挂到同一逻辑上。

## 5. 状态源与职责边界

### 5.1 Rust Core

Rust Core 负责：

- 检测当前 `Sink` 是否自然播放结束。
- 根据当前 `playMode` 决定下一步动作。
- 更新 `PlaybackState`。
- 更新 `Playlist.currentIndex`。
- 广播播放状态和播放列表变化事件。

### 5.2 Frontend

前端负责：

- 订阅 Rust 广播的状态事件。
- 收到事件后刷新本地快照和界面展示。
- 保留低风险的轮询兜底，用于位置更新和事件漏收场景。

前端不负责：

- 判断自然播放结束后是否该切歌。
- 根据播放模式自行推导下一首。

## 6. 自动连播处理链路

### 6.1 播放维护入口

Rust 需要提供一个轻量的“播放维护 tick”入口，职责只有两个：

1. 同步当前播放位置。
2. 检查当前 `Sink` 是否已经自然结束。

如果未结束，只更新位置并返回。

如果已结束，则进入统一的“播放结束处理器”。

### 6.2 播放结束处理器

处理器执行顺序固定为：

1. 读取当前 `PlaybackState`
2. 读取当前 `Playlist` 和当前曲目索引
3. 根据 `playMode` 计算目标动作
4. 执行 `play next`、`replay current` 或 `stop`
5. 更新状态快照
6. 广播事件

所有自然切歌必须通过这条路径完成，不允许在不同命令分支中各自实现一套结束逻辑。

## 7. 播放模式规则

### 7.1 顺序播放 `sequence`

- 正常情况下推进到下一首。
- 如果当前已经是最后一首，则停止播放。
- 停止时：
  - `PlaybackState.isPlaying = false`
  - 保留当前最后一首为当前选中项
  - 不自动跳回第一首

### 7.2 列表循环 `repeat_all`

- 正常情况下推进到下一首。
- 如果当前已经是最后一首，则回到第一首并立即播放。

### 7.3 单曲循环 `repeat_one`

- 自然播放结束后重新播放当前曲目。
- 不修改 `currentIndex`。

### 7.4 随机播放 `shuffle`

- 当播放列表中有多首可播放曲目时，从“除当前曲目外”的候选集中选择下一首。
- 当仅有一首可播放曲目时，行为等同于 `repeat_one`。
- 缺失或不可播放曲目不应进入自动连播候选集。

## 8. 事件模型

为了避免事件过碎，本轮只引入两类统一事件：

- `playback_state_changed`
- `playlist_changed`

事件要求：

- 自然切歌时两个事件都要发。
- 自然停止且索引未变化时，至少要发送 `playback_state_changed`。
- 事件 payload 使用当前完整快照，而不是只发送局部 diff。

这样做的原因是：

- 主窗口、迷你播放器和未来托盘联动都能消费同一事件模型。
- 减少前端事件组合和状态推导错误。

## 9. 前端刷新策略

### 9.1 主窗口

主窗口需要：

- 订阅 `playback_state_changed`
- 订阅 `playlist_changed`
- 在事件到达时刷新当前播放曲目、播放按钮状态、进度和播放模式展示

### 9.2 迷你播放器

迷你播放器需要接入相同的数据来源，而不是继续使用静态标题或只读轮询快照。

自动切歌后至少要同步：

- 曲目标题
- 歌手/专辑文案
- 播放/暂停按钮状态

### 9.3 轮询兜底

当前已有前端轮询不在本轮删除。

保留原则：

- 位置刷新继续轮询，避免为单一进度条单独设计新事件。
- 如果事件偶发丢失，轮询仍能在短时间内纠正状态。

## 10. 数据一致性要求

以下状态必须视为同一个原子更新结果：

- `PlaybackState.trackId`
- `PlaybackState.isPlaying`
- `PlaybackState.positionMs`
- `PlaybackState.durationMs`
- `Playlist.currentIndex`

不允许出现以下中间态：

- `currentIndex` 已移动，但 `trackId` 仍指向旧曲目
- 前端已显示新曲目，但 Rust 的当前索引仍停留在旧位置
- 顺序播放停止后，界面错误显示为仍在播放

## 11. 影响文件

本轮预计主要涉及：

- `src-tauri/src/services/playback.rs`
  - 自然播放结束检测
  - 结束处理器
  - 播放模式推进逻辑
- `src-tauri/src/services/playlist.rs`
  - 当前索引推进与随机候选辅助逻辑
- `src-tauri/src/state.rs`
  - 事件广播辅助能力
- `src-tauri/src/lib.rs`
  - 命令或状态同步入口调整
- `src/App.tsx`
  - 主窗口事件订阅与状态刷新
- `src/main.tsx`
  - 迷你播放器事件订阅与状态刷新
- `src/shared/tauri.ts`
  - 前端事件订阅桥接

## 12. 测试策略

### 12.1 Rust 单元测试

至少覆盖：

- `sequence` 在中间曲目会推进
- `sequence` 在最后一首会停止
- `repeat_all` 在最后一首回到第一首
- `repeat_one` 保持当前索引并重播
- `shuffle` 在多曲目时不直接重复当前曲目
- 缺失或不可播放曲目不会被自动连播选中

### 12.2 Frontend 测试

至少覆盖：

- 主窗口收到 `playback_state_changed` 后更新播放按钮状态
- 主窗口收到 `playlist_changed` 后更新当前曲目展示
- 迷你播放器在自动切歌后更新标题和播放状态

## 13. 风险与约束

主要风险：

- 当前前端已存在轮询逻辑，新增事件后要避免双路径相互覆盖造成闪烁。
- `rodio::Sink` 的结束检测必须区分“自然结束”与“用户主动停止/切歌”。
- 随机模式如果直接依赖简单随机，测试必须避免脆弱的概率断言。

约束：

- 本轮不引入新的外部播放库。
- 不拆现有多窗口架构。
- 不改变应用当前公开命令集合的整体风格。

## 14. 验收标准

完成后应满足：

- 本地歌曲自然播完后，播放器能按 `playMode` 自动推进或停止。
- 顺序播放在最后一首结束时停止，不回到第一首。
- 列表循环、单曲循环、随机播放都按本设计规则工作。
- 主窗口和迷你播放器能在自然切歌后自动刷新，不依赖用户手动操作。
- `npm test`、`npm run build`、`cargo test --manifest-path src-tauri/Cargo.toml` 通过。
