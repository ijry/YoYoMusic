# 悠悠乐听设计规格

**日期:** 2026-07-05
**项目:** 悠悠乐听音乐播放器
**定位:** 基于 Tauri 的跨平台新一代桌面音乐播放器
**技术路线:** Tauri 2 + React + TypeScript + Rust Core
**目标平台:** Windows、macOS、Linux

## 1. 目标与范围

悠悠乐听是一款参考千千静听经典体验的新一代桌面音乐播放器。首版采用现代主界面与怀旧经典皮肤能力结合的方式，面向本地音乐播放场景，同时允许联网补全歌词、封面和元数据。

首版范围包括：

- 本地音频文件播放。
- 手动添加文件和文件夹到播放列表。
- 播放、暂停、上一首、下一首、进度控制、音量、静音和播放模式。
- 歌词、桌面歌词、封面、标签编辑和均衡器。
- 至少三种音乐可视化模式。
- 自定义新皮肤包格式，支持导入、切换、预览和基础校验。
- 系统托盘、全局快捷键、迷你模式、小窗置顶。
- 允许接入第三方公开服务补全歌词、封面和元数据。

首版明确不包含：

- 本地音乐库扫描和歌手/专辑分类库视图。
- 在线曲库播放。
- 老皮肤格式强兼容。
- 开放式插件平台。
- 用户自定义布局引擎。

## 2. 技术路线选择

采用 Tauri 2 + React/TypeScript 前端 + Rust 播放服务核心。

Rust Core 负责音频播放内核、播放状态机、播放队列、系统托盘、全局快捷键、窗口生命周期、文件访问、元数据读写、联网补全调度和持久化。React 前端负责主界面、皮肤渲染、歌词排版、可视化展示、播放列表交互、标签编辑交互和设置界面。

选择该路线的原因：

- 桌面集成能力更稳定，适合托盘、全局快捷键、透明歌词窗、迷你模式和置顶窗口。
- 播放状态集中在 Rust，多个窗口只订阅状态，避免状态分裂。
- 可视化和皮肤渲染放在前端，迭代速度更快。
- 联网补全和皮肤包校验可以通过服务层抽象，后续扩展成本可控。

## 3. 整体架构

系统采用 Rust 主进程核心和 React 多窗口前端结构。

### 3.1 Rust Core

Rust Core 是应用的状态真相源，负责：

- 音频播放、暂停、切歌、seek、音量、静音和均衡器。
- 播放列表增删改查、排序和播放模式。
- 系统托盘和全局快捷键。
- 主窗口、桌面歌词窗口、迷你播放器窗口的生命周期管理。
- 本地文件访问、音频标签读取和标签写回。
- 本地歌词、在线歌词、封面和元数据补全调度。
- 设置、播放列表、缓存和窗口状态持久化。
- 向前端广播播放状态、进度、歌词、可视化帧和错误事件。

### 3.2 React 主窗口

主窗口包含：

- 应用标题区、窗口控制、皮肤切换和设置入口。
- 播放列表区，支持当前列表、拖拽排序、删除、清空和定位当前播放。
- 当前歌曲信息、封面、进度条和主控制按钮。
- 歌词、可视化、标签编辑和均衡器切换区。
- 音量、播放模式、桌面歌词开关、迷你模式入口和置顶入口。

默认布局以单主窗高信息密度为原则，保留经典播放器的效率感，同时使用现代视觉层级。

### 3.3 Overlay Windows

应用包含两个独立辅助窗口：

- `Desktop Lyrics Window`：透明无边框桌面歌词窗口，支持置顶、穿透点击切换、字体、颜色、阴影、描边、双语行距和透明度配置。
- `Mini Player Window`：轻量迷你播放器窗口，保留封面缩略图、歌名、播放控制、简版进度和音量或静音入口，支持置顶、拖动和贴边停靠。

辅助窗口只消费 Rust Core 广播的状态，不维护独立播放状态。

## 4. 模块边界

### 4.1 Frontend Feature Modules

- `player-ui`：播放控制、进度、音量和播放模式。
- `playlist-ui`：列表展示、拖拽排序、批量移除和当前项高亮。
- `lyrics-ui`：主窗歌词、滚动定位和桌面歌词同步展示。
- `visualization-ui`：频谱柱、波形、环形脉冲等可视化模式。
- `skin-ui`：主题变量、背景图、皮肤包切换和预览。
- `settings-ui`：快捷键、联网补全、窗口行为和缓存设置。
- `tag-editor-ui`：标题、歌手、专辑、封面等标签编辑交互。
- `equalizer-ui`：均衡器开关、预设和频段参数交互。

### 4.2 Core Service Modules

- `playback-engine`：播放、暂停、切歌、seek、音量、静音和均衡器。
- `playlist-service`：播放列表增删改查、排序和播放模式。
- `metadata-service`：读取标签、封面和写回标签。
- `lyrics-service`：本地歌词解析、在线歌词匹配和时间轴同步。
- `artwork-service`：封面读取、缓存和远程补全。
- `skin-service`：导入皮肤包、校验 manifest、解压和应用。
- `window-service`：主窗、歌词窗、迷你窗、置顶和显示状态协调。
- `integration-service`：前后端命令和事件桥接。
- `settings-service`：配置读取、写入和迁移。

## 5. 状态与数据流

播放状态只在 Rust Core 中维护。前端发送用户意图，Rust Core 执行命令后广播新状态。

典型命令包括：

- `play(trackId)`
- `pause()`
- `togglePlay()`
- `next()`
- `previous()`
- `seek(positionMs)`
- `setVolume(value)`
- `setMuted(value)`
- `setPlayMode(mode)`
- `applySkin(skinId)`
- `openMiniPlayer()`
- `toggleDesktopLyrics()`

典型事件包括：

- `playbackChanged`
- `positionTick`
- `trackChanged`
- `volumeChanged`
- `lyricsUpdated`
- `visualizationFrame`
- `skinChanged`
- `windowStateChanged`
- `appError`

主窗口、迷你窗口、桌面歌词窗口、托盘菜单和全局快捷键都走同一套命令和事件链路，不允许旁路控制播放状态。

## 6. 本地导入与播放列表

用户可以从主窗口添加单个文件或整个文件夹。Rust Core 负责读取文件路径、基础标签、时长和封面摘要，并生成 `Track` 记录加入当前播放列表。

首版不做音乐库扫描，所以播放列表是唯一曲目组织单位。可以支持多个播放列表，但不提供歌手、专辑或目录库视图。

如果文件不可播放，曲目标记为失败，列表保留该项，播放时自动尝试下一首并提示原因。如果文件丢失，列表中显示失效状态，不自动删除。

## 7. 歌词、封面与联网补全

歌词和封面获取优先级：

1. 音频文件内嵌标签。
2. 同目录本地文件。
3. 本地缓存。
4. 在线匹配。

歌词服务在切歌后异步执行，不阻塞播放。在线补全成功后写入应用缓存，不默认回写原音频文件。标签编辑只有在用户显式保存时才写回音频文件。

联网补全允许接入第三方公开服务。补全源需要通过 provider 接口封装，避免服务实现直接散落到 UI 或播放核心中。

## 8. 可视化

播放引擎输出音频分析数据，Rust Core 统一采样或 FFT 预处理，再以节流后的帧事件发送给前端。

前端至少提供三种可视化模式：

- 频谱柱。
- 波形。
- 环形脉冲。

可视化切换只影响前端渲染，不影响播放稳定性。可视化更新必须节流，并尊重系统减弱动画设置。

## 9. 皮肤系统

皮肤采用自定义新格式，不兼容老播放器皮肤格式。

皮肤包至少包含：

- `manifest.json`：名称、作者、版本、适配应用版本、支持窗口类型和资源声明。
- `theme.json`：颜色变量、字体、间距、圆角、阴影和透明度。
- `assets/`：背景图、按钮图、装饰图、封面遮罩和可视化贴图。
- `layouts/optional.json`：有限度布局偏好，例如封面显示风格、面板背景模式和歌词窗默认样式。

允许皮肤控制：

- 颜色。
- 背景。
- 字体。
- 按钮材质。
- 边框。
- 面板风格。
- 视觉密度。
- 歌词窗默认样式。
- 可视化配色。

不允许皮肤控制：

- 业务逻辑。
- 窗口权限。
- 播放行为。
- 数据结构。
- 任意组件重排。

业务组件不直接读取具体皮肤资源路径，只读取主题变量和皮肤能力声明。

## 11. 存储方案

配置与轻量状态使用 Tauri Store 或等价 JSON 配置持久化。

皮肤包、歌词缓存、封面缓存和联网补全结果放在应用数据目录。播放列表和会话数据首版使用 JSON 存储。由于首版不做音乐库扫描，数据量可控，暂不引入数据库。

持久化内容包括：

- 最近播放列表。
- 窗口状态。
- 当前皮肤。
- 音量。
- 播放模式。
- 均衡器参数。
- 歌词偏移。
- 最近联网补全结果。

应用重启后默认恢复上次会话。如果某些文件丢失，只标记为失效曲目，不让整个播放列表崩溃。

## 12. 标签编辑策略

用户在标签编辑面板修改标题、歌手、专辑或封面时，先更新内存态与界面预览。点击保存后由 Rust Core 调用元数据写回流程。

写回失败时保留临时编辑结果，并提示明确原因，例如文件不可写、格式不支持或权限不足。

## 13. 错误处理

- 音频文件不可播放：曲目标记为失败，列表保留该项，自动尝试下一首，并提示原因。
- 文件丢失：列表中显示失效状态，不自动删除。
- 标签读取失败：使用文件名作为标题，其他字段留空。
- 标签写回失败：保留编辑草稿，提示不可写、格式不支持或权限不足。
- 联网补全失败：不影响播放，显示未找到或请求失败，允许重试。
- 皮肤包无效：导入失败，不改变当前皮肤，提示 manifest、资源缺失或版本不兼容。
- 全局快捷键冲突：该快捷键禁用并提示用户重新设置。
- 桌面歌词窗口异常：只关闭歌词窗，不影响主播放器。

## 14. 非功能约束

- 跨平台：Windows、macOS、Linux 都必须可运行。
- 平台差异：路径、窗口透明、置顶、全局快捷键和桌面歌词穿透需要做平台差异封装。
- 性能：可视化和歌词同步不能影响播放主线程，频谱推送必须节流。
- 可靠性：主播放状态只在 Rust Core 保持，窗口关闭或打开不应导致播放中断。
- 可恢复：应用异常退出后尽量恢复上次列表和播放上下文。
- 可测试：播放队列、皮肤包校验、歌词解析和状态同步逻辑必须能独立测试。
- 可扩展：联网补全源抽象成 provider 接口，首版只接有限公开服务。
- 可访问性：键盘导航完整，焦点清晰，动效尊重系统减弱动画设置。

- 皮肤变量应用。

### 15.4 手工跨平台验收

- Windows、macOS、Linux 分别验证启动、播放、托盘、窗口置顶、透明歌词窗和快捷键。
- 对桌面歌词透明和穿透这类平台差异功能，允许平台能力降级提示。

## 16. 实现阶段建议

虽然产品设计一次性确定，工程实现仍然分阶段推进。

1. 阶段一：Tauri 项目、主窗口、播放列表和基础播放控制。
2. 阶段二：歌词、封面、标签、缓存和联网补全。
3. 阶段三：皮肤包、可视化和均衡器。
4. 阶段四：桌面歌词、迷你模式、托盘、全局快捷键和跨平台验收。

每个阶段都服务于同一份完整设计，不缩减最终产品目标。
