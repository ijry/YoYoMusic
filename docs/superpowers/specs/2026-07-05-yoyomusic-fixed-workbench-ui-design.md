# 悠悠乐听固定工作台 UI 改造设计

**日期:** 2026-07-05
**项目:** 悠悠乐听音乐播放器
**主题:** 主窗口从网页式滚动壳升级为固定桌面播放器工作台

## 1. 背景

当前主窗口已经具备播放器核心功能，但视觉和布局仍偏“网页页面”：

- 外层使用 `min-height` 和居中大面板，窗口高度不足时整窗会滚动。
- 播放列表、当前播放、功能面板和控制条没有形成清晰的桌面软件工作区。
- 底部控制区会随页面内容移动，缺少播放器应有的“始终可控”感。
- 信息密度偏松散，整体看起来简陋，不像专业桌面音乐软件。

本轮只做主窗口 UI/布局收口，不扩展新的播放能力。

## 2. 目标

本轮目标：

- 主窗口在桌面尺寸下固定为 `100vh` 工作台，禁止 `body`/整窗滚动。
- 播放列表和右侧功能面板各自拥有内部滚动，不影响整体窗口。
- 底部播放控制条固定在主窗口底部，始终可见。
- 从“大卡片网页”改成更像桌面音乐播放器的紧凑专业界面。
- 保留现有青绿科技感和皮肤 token，不引入新的设计系统依赖。

## 3. 非目标

本轮不做：

- 新增播放功能或改动 Rust 播放逻辑。
- 重写皮肤包格式。
- Winamp/千千静听 1:1 复古皮肤系统。
- 移动端优先布局。
- 虚拟列表或大型播放库管理。
- 新增图标库、组件库或字体下载依赖。

## 4. 设计方向

采用 **固定三栏桌面工作台 + 底部硬件控制条**。

窗口结构：

1. 顶部紧凑标题栏
2. 中部三栏工作区
3. 底部固定播放控制条

三栏工作区：

- 左栏：播放列表和导入操作
- 中栏：当前播放核心区、封面唱片和重点可视化
- 右栏：歌词、标签、均衡器、皮肤、设置等功能面板

这个方向比“只加 `overflow: hidden`”更可靠，因为它明确了哪些区域应该滚动、哪些区域必须固定。

## 5. 布局规格

### 5.1 根布局

`html`、`body`、`#root` 和 `.app-shell` 必须占满视口：

- `height: 100%`
- `overflow: hidden`
- `min-width: 320px`

`.app-shell` 不再用居中页面式 `min-height` 布局，而是作为完整窗口背景容器。

### 5.2 主窗口壳 `.chrome`

`.chrome` 是主桌面软件外壳：

- 高度：`height: calc(100vh - 24px)`
- 宽度：`min(1280px, calc(100vw - 24px))`
- 使用 CSS Grid 分三行：标题栏、工作区、控制条
- 行定义建议：`auto minmax(0, 1fr) auto`
- 自身 `overflow: hidden`

这样标题栏和控制条始终固定，中间工作区消化剩余高度。

### 5.3 顶部标题栏

标题栏应更紧凑：

- 应用名仍为 `悠悠乐听`
- 副标题可保留，但字号降低
- 窗口操作按钮右对齐
- 高度控制在约 `72px` 内

目标是“播放器工具栏”，不是 landing page hero。

### 5.4 中部工作区

`.workspace` 改为固定高度三栏 Grid：

- 左栏：`minmax(260px, 0.85fr)`
- 中栏：`minmax(320px, 1.15fr)`
- 右栏：`minmax(280px, 0.95fr)`
- 高度：`minmax(0, 1fr)`
- `overflow: hidden`

左栏播放列表和右栏功能面板内部滚动；中栏主要内容随高度压缩。

### 5.5 左栏播放列表

播放列表应像桌面软件列表：

- 播放列表面板高度填满工作区。
- 标题和导入按钮固定在面板顶部。
- `.track-list` 使用 `overflow: auto`。
- 当前曲目高亮更明显，例如左侧色条 + 背景渐变。
- 长标题保持单行省略。

播放列表过长时，只滚动歌曲列表，不滚动整个窗口。

### 5.6 中栏当前播放

当前播放区应聚焦“正在播放”：

- 封面唱片尺寸使用 `clamp()` 随高度收缩。
- 曲名和歌手放在封面下方或右侧，保持层级清晰。
- 默认展示一个主可视化区域，使播放器不显空。
- 低高度窗口时减少间距和封面尺寸，不触发整窗滚动。

现有功能面板不再全部堆在中栏，避免当前播放区变成杂乱页面。

### 5.7 右栏功能面板

右栏承载功能标签和当前功能内容：

- 标签栏固定在右栏顶部，可换行但不撑破窗口。
- 功能内容容器 `min-height: 0; overflow: auto`。
- 歌词、标签、均衡器、皮肤、设置共用统一面板视觉。

功能内容过长时，只滚动右侧面板。

### 5.8 底部播放控制条

播放控制条固定在 `.chrome` 第三行：

- 采用硬件播放器面板风格。
- 左侧：上一首、播放/暂停、下一首、静音。
- 中间：进度条和时间。
- 右侧：音量和播放模式。
- 不随播放列表或功能内容滚动。

按钮应更像桌面控件，保留可见 focus 状态和 hover 反馈。

## 6. 响应式规则

主目标尺寸：

- 主要保证 `1024x700` 及以上桌面窗口体验。
- `1280x800` 及以上应呈现完整三栏工作台。

窄窗口规则：

- 小于约 `980px` 时，中部改为两栏：播放列表 + 当前播放/功能堆叠区。
- 小于约 `760px` 时允许主内容内部纵向排布，但仍尽量避免 `body` 滚动。
- 如果极小窗口无法容纳全部控件，优先让 `.chrome` 内部工作区滚动，而不是 `body` 滚动。

本轮不以手机竖屏作为主要验收目标。

## 7. 视觉规格

保留现有青绿/浅蓝音乐科技感，但提高专业度：

- 背景：深色渐变 + 低透明噪点/光斑感。
- 面板：玻璃金属质感，边框更清晰。
- 控制条：更高对比、更像硬件控制面板。
- 字体：继续使用现有 Poppins / Noto Sans SC / Microsoft YaHei 栈，不新增网络字体依赖。
- 动效：只保留唱片旋转和轻微 hover，不引入滚动动效。

避免：

- 页面式大标题占用过多高度。
- 纯靠大片空白制造“高级感”。
- 整窗滚动。
- Hover 位移导致布局跳动。

## 8. 可访问性与交互

必须保留：

- 现有按钮文本和 aria label。
- `:focus-visible` 键盘焦点。
- `prefers-reduced-motion` 降低动画。
- 输入控件可键盘操作。

新增约束：

- 内部滚动区应有足够可见高度，不能出现 20px 级别的不可用滚动窗。
- 控制条在任何正常桌面高度都必须可见。
- 不隐藏错误提示；错误提示应占据标题栏下方或工作区顶部的固定区域。

## 9. 影响文件

预计涉及：

- `src/App.tsx`
  - 调整主窗口 DOM 分区，让中栏当前播放和右栏功能面板分离。
- `src/styles/app.css`
  - 根布局固定视口。
  - 三栏工作区。
  - 内部滚动区域。
  - 控制条固定底部。
  - 响应式规则。
- `src/styles/theme.css`
  - 仅在需要统一控制条、滚动区或面板色值时补充主题变量；不改变现有变量语义。
- 测试文件
  - 增加或调整主窗口渲染测试，确保主要区域仍可访问。

## 10. 测试策略

自动化测试：

- `npm test`
- `npm run build`

布局证据：

- 搜索确认根布局和主要滚动容器：
  - `overflow: hidden`
  - `.track-list`
  - `.feature-content`
  - `.player-controls`

人工验收：

- 在约 `1280x800` 打开主窗口，整窗不滚动。
- 在约 `1024x700` 打开主窗口，控制条仍可见。
- 导入多首歌曲后，只滚动播放列表。
- 切换到皮肤/设置/均衡器等长内容后，只滚动右侧功能面板。
- 播放控制条始终可见。

## 11. 风险与约束

主要风险：

- 当前 `App.tsx` 同时承担状态、布局和功能渲染，DOM 调整要避免破坏现有测试。
- CSS Grid 固定高度如果没有正确设置 `min-height: 0`，内部滚动会失效。
- 过度压缩可能让功能面板可用性下降。

约束：

- 不引入新依赖。
- 不改变 Tauri/Rust 命令。
- 不改变已完成的自动连播事件模型。
- 不影响迷你播放器和桌面歌词路由。

## 12. 验收标准

完成后应满足：

- 主窗口整体不出现 `body`/整窗滚动。
- 播放列表长内容只在左栏内部滚动。
- 功能面板长内容只在右栏内部滚动。
- 底部播放控制条始终固定可见。
- 视觉明显比当前版本更像桌面播放器，而不是网页页面。
- `npm test` 和 `npm run build` 通过。
